# Validation Scripts Guide

The phase-audit skill requires a passing validation script before marking any behavioral phase COMPLETE. These live at `scripts/validate_*` and are **meant to be generated by the LLM** as part of each phase — every project is different, so the scripts will be different.

## What makes a good validation script

A validation script should **prove the phase works**, not just check that files exist. Good scripts:

- **Test actual behavior** — import the module, call the function, hit the endpoint, query the database
- **Assert on outcomes** — not "file exists" but "function returns expected result"
- **Run without manual setup** — no environment variables to set, no services to start (or the script starts them)
- **Exit 0 on success, non-zero on failure** — the phase-audit checks the exit code
- **Print what they're checking** — so the audit report is readable

## Example prompts for generating validation scripts

After completing a phase, prompt the LLM:

> "Write a validation script at `scripts/validate_phase_01.sh` that proves Phase 1 works. It should test [the API endpoints / the database schema / the CLI commands] we just built. Run it and confirm it passes."

> "Before we mark this phase complete, write `scripts/validate_phase_03.sh` that verifies the auth middleware rejects unauthenticated requests and passes authenticated ones. Execute it."

> "Generate a validation script for this phase. It should import our modules directly and assert the core behavior works — not just check that files exist."

## What to avoid

- **File-existence checks** — `test -f src/foo.sh` proves nothing
- **Syntax-only checks** — checking a file parses doesn't test behavior
- **Mocks of the thing being validated** — if you're validating the database layer, hit the real database

## Non-behavioral phases

Documentation-only or planning phases are exempt from validation scripts. The phase-audit marks these as `EXEMPT` automatically.
